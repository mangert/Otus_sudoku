## 1. Реализация алгоритма

### 1.1. Основная идея

Судоку сводится к **задаче точного покрытия (Exact Cover)**. Каждая клетка, строка, столбец и блок — это ограничение (столбец матрицы). Каждый возможный вариант «поставить цифру X в клетку (r,c)» — это строка матрицы, покрывающая ровно 4 ограничения (тут подумать - а нельзя ли сократить? Они частично связаны...).

Матрица получается размера **729 × 324**. Она очень разреженная (в каждой строке ровно 4 единицы), поэтому хранить её в виде двумерного массива нельзя - будет огромный перерасход памяти и времени (тут опять же на подумать - может быть действительно ограничения как-то упаковать?).

### 1.2. Структура данных Dancing Links (DLX)

Реализовать класс `DancingLinksMatrix`, внутри которого:

- **Узел (struct Node)**:
  - указатели `left`, `right`, `up`, `down`
  - указатель `column` на «голову» столбца (или `nullptr` для узлов-заголовков)
  - для узлов-заголовков столбцов: `size` (количество единиц в столбце), `name` (индекс ограничения)

- **Горизонтальный циклический список всех заголовков столбцов** (чтобы перебирать столбцы и выбирать минимальный по `size`).

- **Вертикальные циклические списки** внутри каждого столбца (включая сам узел-заголовок).

**Ключевые операции:**

- `cover(ColumnNode* col)` — удаляет столбец из заголовков и все строки, в которых есть единица в этом столбце, из других столбцов. Это «запрещает» варианты, конфликтующие с выбранным.
- `uncover(ColumnNode* col)` — восстанавливает удалённое (обратный порядок действий). Благодаря циклическим спискам и сохранению указателей, восстановление делается без дополнительной памяти.

**Вспомогательные операции:**

- подумать, что нужно, уж точно `empty()`, может еще что-то

**Поиск решения (рекурсивный или итеративный - подумать. И что возвращаем):**

1. Если все столбцы удалены решение найдено.
2. Выбрать столбец с **минимальным `size`** (эвристика, резко ускоряет перебор).
3. Перебрать все строки в этом столбце (по `down`):
   - Выбрать эту строку (запомнить её узлы)
   - `cover` для всех столбцов, в которых есть единицы в этой строке
   - Рекурсивный вызов
   - Если решение найдено — вернуть успех
   - Иначе `uncover` для всех тех же столбцов (откат) и попробовать следующую строку
4. Вернуть `false` (нет решения)

### 1.3. Класс судоку (`SudokuSolver`)

- **Построение матрицы** (один раз при инициализации):
  - Создать 324 столбца (4 группы ограничений: клетки, строки, столбцы, блоки. Тут тоже подумать).
  - Создать 729 строк — по одной на каждую тройку `(row, col, digit)`.
  - Для каждой строки определить, в какие 4 столбца ставить единицы.
  - Добавить строку в матрицу DLX.

- **Учёт начальных условий**:
  - Для каждой заданной клетки (с фиксированной цифрой) «выбрать» соответствующую строку принудительно (вызываем `cover` для её столбцов до начала поиска). Это сужает пространство поиска.

- **Запуск решения**:
  - Вызывать поиск на подготовленной матрице.
  - Если решение найдено — по сохранённым выбранным строкам восстанавивить итоговую доску 9×9.
  - Если нет — вывести, что судоку неразрешимо.

## 2. Тестирование

### 2.1. Проверка корректности (точность)

Для решённой доски **обязательно** проверить:

- В каждой строке сумма цифр = 45 (1+...+9) **и** все цифры различны.
- В каждом столбце — аналогично.
- В каждом квадрате 3×3 — аналогично.

Это простая и надёжная валидация, не требующая сравнения с эталоном.

Для **нерешаемых** судоку:
- Возьмём заведомо нерешаемые головоломки (например, с двумя одинаковыми цифрами в одной строке).
- Убедимся, что алгоритм возвращает «решения нет» за конечное время (без зависаний).

### 2.2. Производительность

Измерить (замеры в debug-режиме, под #IFDEF DEBUG):

- **Время решения** (в микросекундах через `<chrono>`).
- **Количество итераций** (глубина рекурсии + число откатов) — добавить счётчик.
- **Количество вызовов `cover` / `uncover`**.

**Тестовые наборы**:
- Лёгкие судоку (много подсказок)
- Средние
- Сложные (например, *Al Escargot*, *Platinum Blonde*)
- Пустая доска (найти все решения не будем, но одно — быстро)

Будет произведено сравнение с **наивным бэктрекингом** (без эвристик) — на тех же данных, чтобы показать выигрыш DLX.

### 2.3. На что сделан упор

**Приоритет 1 — корректность.** Решение должно быть всегда правильным, если оно существует.

**Приоритет 2 — производительность** за счёт:
- Эвристики выбора столбца с минимальным размером.
- Эффективного хранения разреженной матрицы (списки вместо массива).
- Минимизации копирований данных (работа с указателями).
